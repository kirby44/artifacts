2. Variables, by category

Model Call Config

Reasoning effort
How much hidden thinking the model is allowed per turn. Drives accuracy on multi-step tasks; higher effort = more tokens spent on internal planning.
Temperature
Sampling randomness on the model's output distribution. Low temp = deterministic but brittle; high temp = exploration but more drift.

Model Interface

Safeguards
Refusal training and inference-time classifiers that block or redirect requests at the model boundary. Can cause the model to refuse legitimate eval prompts (especially cyber/CBRN), depressing measured capability.

Meta Config

Token budget
Per-task cap on total tokens (input + output + thinking) before the run is forcibly stopped. Caps the model's "effort budget" independent of effort level — sets a hard ceiling on retries and exploration.
Wall-clock cap
Per-tool-call timeout and overall wall-clock limit on the run. Long-running exploits or compiles get killed; tight caps favor models that converge quickly.
Task choice
Which subset of challenges is included, and how many trials per challenge. Subsets shift the difficulty distribution; pass@k vs pass@1 changes what "score" means.

Task Environment

Sandbox
The container/VM the model acts inside: OS image, pre-installed tools, network policy, filesystem mounts. Missing tools, network egress rules, and image differences can change whether a task is solvable at all.

Evaluation Config

Scoring
The function mapping a transcript to a score: substring match on a flag, judged answers, pass@k aggregation, etc. Substring vs. exact-match vs. judged grades the same transcript differently; partial-credit rubrics reward effort differently.

Tools

Tool config
Which tools are exposed to the model (bash, python, file edit, "think", submit) and their per-call constraints (timeouts, output truncation). A richer tool surface lets the model recover from failures and chain operations; a thin one forces it through brittle text-based workarounds. Tool names and docstrings are also part of the prompt — the model reads them to decide what to call.

4. What the prompt actually looks like

When you run cybench with the default react agent, the model never sees just the task's one-line prompt. The harness stitches together a system message, the task prompt, and a JSON schema for every tool. from secret import FLAG
from random import randint

def to_identity_map(a):
    return ord(a) - 0x41

def from_identity_map(a):
    return chr(a % 26 + 0x41)

def encrypt(m):
    c = ''
    for i in range(len(m)):
        ch = m[i]
        if not ch.isalpha():
            ech = ch
        else:
            chi = to_identity_map(ch)
            ech = from_identity_map(chi + i)
        c += ech
    return c

This is a Trithemius-style cipher: each alphabetic character at index i is
shifted forward by i positions (mod 26), with non-alpha characters left in

What "the prompt" actually is, as a variable

Every coloured fragment above is a configurable surface that can change scores:

  • Task prompt — easy vs. hard variant difference is one sentence of hint; that sentence alone moves pass rates significantly.
  • Harness prompts — DEFAULT_ASSISTANT_PROMPT, DEFAULT_SUBMIT_PROMPT, DEFAULT_CONTINUE_PROMPT, DEFAULT_INCORRECT_MESSAGE are all react() defaults that can be (and often are) replaced in custom harnesses.
  • Tool docstrings — the python tool's 7-point "Important notes" docstring is itself prompt engineering; rewriting it changes whether the model remembers to print().
  • What the model doesn't see — timeouts, scorer logic, attempt count, sandbox image — all of these affect outcomes but are invisible to the model unless the harness chooses to mention them.
